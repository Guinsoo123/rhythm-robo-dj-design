# 强化学习舞蹈控制

本文档描述强化学习舞蹈控制的工程化设计（架构、状态/动作/奖励、训练与部署）。**定义、直觉与公式推导**见 [摇滚机器人跳舞原理](README_摇滚机器人跳舞原理.md)。

## 为什么这个问题适合强化学习

让机器人跟摇滚音乐摇头晃脑，看起来像一个动作生成问题，但本质上是一个带安全约束的连续控制问题。

传统脚本可以写成：

```text
每一拍点头一次
每两拍身体下蹲一次
副歌时加大幅度
```

这种方法容易做出第一个 demo，但问题很快出现：

- 机器人站姿被扰动时，脚本不知道怎么恢复平衡。
- 不同 BPM、不同音乐强弱、不同地面摩擦都会影响动作安全。
- 动作幅度一大，关节耦合、机身姿态和足端接触会变复杂。
- 手写规则很难同时满足“好看、跟拍、稳定、省力”。

强化学习的优点是：我们不直接告诉机器人每个关节应该怎么动，而是告诉它“什么样的结果更好”。机器人在仿真中反复尝试，逐步学会在节拍条件下保持平衡并生成动作。

## 新手先理解四个词

| 概念 | 在本项目中的含义 | 直觉 |
| --- | --- | --- |
| 状态 state | 机器人和音乐当前是什么情况 | 机器人站得稳不稳，音乐到第几拍了 |
| 动作 action | 策略每一小步发出的控制指令 | 关节位置目标、速度目标或残差力矩 |
| 奖励 reward | 对当前行为好坏的打分 | 跟上节拍加分，摔倒扣很多分 |
| 策略 policy | 从状态到动作的函数 | 机器人“大脑”：看到状态后决定怎么动 |

一次训练循环可以想成：

```text
机器人观察状态 -> 策略输出动作 -> MuJoCo 推进物理仿真 -> 得到新状态和奖励 -> 更新策略
```

训练不是在真机上乱试，而是在 MuJoCo 中试。只有通过稳定性、安全性和节奏性评估的策略，才允许进入真机小幅度验证。

更完整的物理、音乐、强化学习与 PPO 推导见 [摇滚机器人跳舞原理](README_摇滚机器人跳舞原理.md)。

## 算法架构设计

本节从系统视角说明“摇滚机器人跳舞”在算法层的模块划分、数据流、形式化对象与运行时边界。实现细节与公式推导见原理文档；本文档侧重工程可落地的架构视图。

### 设计目标与硬约束

| 目标 | 架构含义 |
| --- | --- |
| 节奏性 | 策略输入必须包含可微、连续的节拍相位，而非离散 beat 脉冲 |
| 稳定性 | 奖励与安全层双保险；动作以默认站姿残差输出 |
| 可迁移 | 训练侧域随机化 + 部署侧动作裁剪、低通、急停 |
| 可评估 | 离线指标（fall rate、beat alignment）与在线门控分离 |

硬约束在架构上体现为三条不可绕过的链路：

```text
(1) 音乐条件 c_t 只进入策略观测，不直接映射关节
(2) 策略输出 a_t 必须经过 SafetyFilter 才成为电机目标
(3) 真机路径在 Policy 与电机之间增加 ReplayCheck + EmergencyStop
```

### 端到端数据流

```text
                    ┌─────────────────────────────────────────┐
  摇滚音频 x(t) ──► │ AudioFeaturePipeline（离线/在线）        │
                    │  BPM, beat_times, onset, RMS, section   │
                    └──────────────────┬──────────────────────┘
                                       │ music_context c_t
  MuJoCo 传感器 ──► ┌──────────────────▼──────────────────────┐
  q, qdot, IMU,     │  ObservationBuilder                      │
  contact, ...     │  s_t = [s_robot, c_t, a_{t-1}]           │
                    └──────────────────┬──────────────────────┘
                                       │ state s_t
                    ┌──────────────────▼──────────────────────┐
                    │  Policy pi(a|s)  （PPO Actor，部署取均值） │
                    └──────────────────┬──────────────────────┘
                                       │ raw action a_t
                    ┌──────────────────▼──────────────────────┐
                    │  SafetyFilter + ActionScale + LPF      │
                    │  q* = q_default + alpha * clip(a_t)    │
                    └──────────────────┬──────────────────────┘
                                       │ joint targets
                    ┌──────────────────▼──────────────────────┐
                    │  LowLevel PD / 宇树底层关节控制器         │
                    └──────────────────┬──────────────────────┘
                                       │ torques tau
                    ┌──────────────────▼──────────────────────┐
                    │  MuJoCo / 真机 多体动力学 + 接触           │
                    └──────────────────┬──────────────────────┘
                                       │ s_{t+1}, r_t
                    ┌──────────────────▼──────────────────────┐
                    │  RewardEngine + PPO Learner（仅训练）     │
                    └─────────────────────────────────────────┘
```

### 分层模块职责

| 模块 | 输入 | 输出 | 训练/推理 |
| --- | --- | --- | --- |
| AudioFeaturePipeline | 音频波形或预计算特征文件 | `c_t`（相位、能量、BPM 等） | 二者 |
| ObservationBuilder | 仿真/真机传感 + `c_t` + 历史动作 | `s_t` | 二者 |
| Policy | `s_t` | `a_t` 或 `mu_t` | 二者 |
| SafetyFilter | `a_t`, `s_t` | `q*_t` 与是否急停 | 二者（训练也启用，防投机） |
| RewardEngine | `s_t,a_t,s_{t+1},c_t` | 标量 `r_t` | 仅训练 |
| PPO Learner | rollout 轨迹 | 更新后的 `pi, V` | 仅训练 |
| Evaluator | 策略 + 未见歌曲 | 指标与视频 | 离线 |

模块间禁止的依赖（避免后期难维护）：

- RewardEngine 不得反向修改 Policy 参数（奖励塑形只通过 `r_t`）。
- AudioFeaturePipeline 不得读取机器人状态（防止信息泄漏式“作弊”对齐）。
- 真机部署不得跳过 SafetyFilter，即使仿真中曾关闭过。

### MDP 形式化

将舞蹈控制建模为条件 MDP（音乐作为外生条件过程）：

| 符号 | 定义 |
| --- | --- |
| `s_t` | 机器人本体观测 + 音乐条件 + `a_{t-1}` |
| `a_t` | 关节位置残差（第一版），维度 `n_joint` |
| `c_t` | 音乐条件向量，由音频管线生成 |
| `P(s'|s,a,c)` | MuJoCo 推进的动力学（训练时带域随机化） |
| `r_t` | 稳定 + 节拍 + 风格 + 能量匹配 - 惩罚 |
| `gamma` | 折扣因子，通常 0.99 左右 |
| `pi(a|s)` | 策略，部署时用 `mu(s)` |

一步交互：

```latex
a_t ~ pi(.|s_t),  s_{t+1} ~ P(.|s_t,a_t,c_t),  r_t = R(s_t,a_t,s_{t+1},c_t)
```

优化目标（期望折扣回报）：

```latex
J(pi) = E[ sum_{t=0}^{T} gamma^t r_t ]
```

直觉：我们不搜索一条固定关节轨迹，而是搜索一个“看到状态与音乐就输出动作”的函数；扰动、BPM 变化、摩擦变化都由同一 `pi` 处理。

### 控制周期与时序

| 层级 | 典型频率 | 说明 |
| --- | --- | --- |
| 音频特征 | 与仿真步对齐或 50–100 Hz | beat phase 按仿真时间 `t_sim` 查表插值 |
| RL 策略 | 50–200 Hz | 每个 env step 调用一次 `pi` |
| 低层 PD | 500–1000 Hz | MuJoCo 子步或真机驱动器内环 |
| 安全监控 | 与策略同频或更高 | 姿态越界、接触异常、力矩峰值 |

时序上，第 `k` 个控制步：

```text
t_k: 读取传感 -> 构造 s_t -> pi 输出 a_t -> 安全裁剪 -> q*_t
     -> 低层跟踪 -> 物理积分 -> s_{t+1} ->（训练）计算 r_t
```

音乐时间 `t_audio` 与仿真时间 `t_sim` 在训练时通常锁定（同一首歌驱动）；真机播放时需处理声画延迟，部署架构预留 `audio_delay_compensation` 配置项。

### 观测与动作接口（架构契约）

观测向量拼接（与后文“状态设计”一致，此处强调接口）：

```latex
s_t = concat( s_robot_t, c_t, a_{t-1} )
```

```latex
s_robot_t = [ g_proj, omega_base, q, qdot, contact, f_contact, h_base, ... ]
```

```latex
c_t = [ sin(phi_b), cos(phi_b), sin(phi_bar), cos(phi_bar), bpm_norm, o_t, e_t, I_sec, conf_tempo ]
```

动作（残差位置控制）：

```latex
q*_t = q_default + alpha * clip(a_t, a_min, a_max)
```

`alpha` 为 `action_scale`，训练后期逐步增大；真机首测为训练值的 20%–40%。

### 奖励架构（模块化）

RewardEngine 内部按项累加，便于 ablation：

```latex
r_t = w_stab * R_stab + w_beat * R_beat + w_style * R_style + w_energy * R_energy
      - w_safe * P_safe - w_smooth * P_smooth - w_cost * P_cost
```

| 子模块 | 依赖观测量 | 架构作用 |
| --- | --- | --- |
| R_stab | roll, pitch, h, v | 保证 MDP 有“站立”吸引域 |
| R_beat | phi_b, 低维 bob 特征 | 耦合音乐相位与身体节律 |
| R_style | pitch/roll 周期统计 | 摇滚顿挫感，非芭蕾平滑 |
| R_energy | RMS, joint/base 速度 | 强弱随音乐 |
| P_safe | 限位、力矩、冲击 | 防止奖励黑客 |
| P_smooth | ||a_t - a_{t-1}|| | 抑制高频抖动 |

每项权重放在 YAML，训练课程按阶段启用子集（见后文“训练课程设计”）。

### 训练架构 vs 推理架构

| 阶段 | 额外组件 | 说明 |
| --- | --- | --- |
| 并行训练 | VectorEnv × N, PPO RolloutStorage | 数千环境并行采样 |
| 课程学习 | CurriculumScheduler | 按阶段切换奖励子集与随机化强度 |
| 评估 | EvalRunner + 未见歌曲集 | 不更新参数 |
| 真机推理 | OfflineReplay, ActionLPF, EStop | 无 RewardEngine、无梯度 |

推理路径最小集合：

```text
audio features -> s_t -> pi(mean) -> SafetyFilter -> LPF -> robot
```

### 架构级风险与对策

| 风险 | 架构对策 |
| --- | --- |
| 策略利用仿真接触模型漏洞 | 随机化摩擦/弹性 + 真机前冲击指标门禁 |

| 观测维度过大难训 | 第一版 MLP + 手工音乐特征，不上复杂 RNN |

## 音乐如何变成机器人能理解的信号

机器人不能直接理解“这首歌很燃”。我们需要把音乐拆成数值特征。

### 基础音频特征

| 特征 | 含义 | 用途 |
| --- | --- | --- |
| BPM | 每分钟节拍数 | 决定动作基础周期 |
| beat times | 每个拍点的时间 | 用来对齐点头、下蹲、摆动峰值 |
| downbeat | 小节第一拍或重拍 | 用来做更明显的动作强调 |
| onset strength | 新声音出现的强度 | 鼓点、吉他重击时提高动作能量 |
| RMS energy | 音量能量 | 控制整体动作幅度 |
| spectral centroid | 声音亮度 | 可用于区分尖锐吉他和低频鼓点 |
| tempo confidence | 节拍置信度 | 置信度低时动作更保守 |

### 节拍相位

节拍时间点是离散的，但机器人控制是连续的。因此策略不应该只知道“现在是不是 beat”，还应该知道“当前在一个 beat 周期里的位置”。

定义：

```text
beat_phase = 当前时间距离上一个 beat 的比例
```

如果 `beat_phase = 0.0`，表示刚到拍点；如果 `beat_phase = 0.5`，表示在两个拍点中间；如果接近 `1.0`，表示快到下一个拍点。

为了让神经网络更容易学习相位，通常不用单个 0 到 1 的数，而用正弦和余弦：

```text
phase_sin = sin(2 * pi * beat_phase)
phase_cos = cos(2 * pi * beat_phase)
```

这样 `0.99` 和 `0.01` 在数学上是接近的，网络不会误以为它们差很远。

### 音乐条件向量

最终给策略的音乐条件可以是：

```text
music_context = [
  phase_sin,
  phase_cos,
  downbeat_phase_sin,
  downbeat_phase_cos,
  normalized_bpm,
  onset_strength,
  rms_energy,
  section_intensity,
  tempo_confidence
]
```

这里的 `section_intensity` 是段落强弱，比如主歌为 0.4，副歌为 0.8，breakdown 为 0.6。第一版可以用能量曲线自动估计，后续可以用更复杂的音乐结构分析模型。

## 状态设计

策略的输入状态由三部分组成：机器人本体状态、音乐条件、历史动作。

### 机器人本体状态

| 状态项 | 说明 |
| --- | --- |
| base orientation | 机身姿态，通常用四元数或 roll/pitch/yaw 表示 |
| base angular velocity | 机身角速度，用于判断是否正在失稳 |
| projected gravity | 重力在机身坐标系下的方向，常用于腿足机器人控制 |
| joint positions | 每个关节当前角度 |
| joint velocities | 每个关节当前速度 |
| foot contacts | 足端是否接触地面 |
| foot contact forces | 足端接触力，用于限制冲击和打滑 |
| base height | 机身高度，用于实现下蹲/弹跳感 |

### 音乐条件状态

音乐条件告诉机器人“现在应该用什么节奏、什么强度动”。

| 状态项 | 说明 |
| --- | --- |
| beat phase | 当前拍内相位 |
| bar phase | 当前小节内相位 |
| intensity | 当前音乐强度 |
| onset | 当前是否有明显鼓点或重击 |
| bpm | 当前速度 |
| mode id | 可选，动作模式，例如点头、摆动、弹跳 |

### 历史动作

把上一帧动作 `last_action` 输入策略，可以帮助减少抖动。没有历史动作时，策略可能在相邻控制周期输出差异很大的关节目标，造成电机发抖。

## 动作设计

动作空间有三种常见选择。

| 动作类型 | 含义 | 优点 | 风险 |
| --- | --- | --- | --- |
| 关节位置目标 | 策略输出每个关节的目标角度偏移 | 稳定，适合第一版 | 动作响应可能偏软 |
| 关节速度目标 | 策略输出期望速度 | 动作更顺滑 | 需要更小心限速 |
| 力矩目标 | 策略直接输出力矩 | 表达能力强 | 最危险，真机迁移难 |

建议第一版使用“默认站姿 + 策略输出残差”的位置控制：

```text
target_q = default_q + action_scale * policy_output
```

这样即使策略输出接近 0，机器人也会回到稳定站姿。`action_scale` 可以先设置得很小，比如只允许每个关节在安全范围内产生 5% 到 15% 的偏移，再逐步放大。

## 奖励函数设计

奖励函数是整个算法设计的核心。奖励写得好，策略会学出自然又稳定的动作；奖励写得差，策略可能学会钻空子。

总奖励可以拆成：

```text
reward =
  稳定奖励
  + 节拍对齐奖励
  + 舞蹈风格奖励
  + 能量匹配奖励
  - 安全惩罚
  - 动作不平滑惩罚
  - 能耗惩罚
```

### 稳定奖励

稳定是第一优先级。机器人跟不上拍可以接受，摔倒不可以接受。

| 奖励项 | 目的 |
| --- | --- |
| 姿态保持 | 限制 roll/pitch 过大，避免侧翻和前翻 |
| 高度范围 | 允许下蹲和弹跳，但不能塌陷或过高冲击 |
| 足端接触 | 鼓励至少保留稳定支撑，避免无控制离地 |
| 速度限制 | 限制不必要的机身漂移 |

示例：

```text
r_upright = exp(-k1 * (roll^2 + pitch^2))
r_height = exp(-k2 * (base_height - target_height)^2)
r_no_fall = 1 if not fallen else -10
```

新手理解：`exp(-误差)` 的意思是误差越小分越高，误差越大分越接近 0。

### 节拍对齐奖励

我们希望某些动作在 beat 附近达到峰值。例如点头动作可以让 pitch 或“头部虚拟关节”在拍点附近达到向下峰值；弹跳动作可以让机身高度在重拍后出现节奏性变化。

第一版可以定义一个目标节奏曲线：

```text
target_bob = sin(2 * pi * beat_phase)
```

然后让某个身体特征跟它对齐：

```text
r_beat = exp(-k * (observed_bob - target_bob)^2)
```

其中 `observed_bob` 可以是机身高度变化、pitch 角变化或选定关节组合。

更稳的做法是不要强迫每个关节跟节拍，而是奖励“身体整体运动的某个低维指标”和 beat 相位一致。

### 摇滚风格奖励

摇滚音乐常见动作不是芭蕾式平滑，而是有重拍、有顿挫、有身体重心参与。可以设计以下风格指标：

| 指标 | 含义 |
| --- | --- |
| head bang | pitch 方向有周期性点头 |
| body sway | roll/yaw 或左右腿姿态有节奏摆动 |
| crouch bounce | 重拍附近短暂降低再恢复机身高度 |
| accent | downbeat 时动作幅度略大 |

如果机器人没有真实头部关节，可以把“头部动作”抽象为机身 pitch 或上身 pitch。如果使用四足机器人，头部可理解为机身前部视觉上的点头感。

### 能量匹配奖励

音乐强的时候动作应该更明显，音乐弱的时候动作应该收敛。可以把动作幅度和音乐能量关联起来：

```text
target_motion_energy = min_energy + gain * music_energy
observed_motion_energy = norm(base_angular_velocity) + c * norm(joint_velocity)
r_energy = exp(-k * (observed_motion_energy - target_motion_energy)^2)
```

注意不要鼓励无限加速，所以必须和能耗、安全惩罚一起使用。

### 安全惩罚

| 惩罚项 | 防止什么 |
| --- | --- |
| joint limit penalty | 关节撞限位 |
| torque penalty | 电机过载 |
| velocity penalty | 关节高速抖动 |
| action rate penalty | 相邻动作突变 |
| foot slip penalty | 足端接触地面时滑动 |
| contact impact penalty | 落地冲击过大 |

示例：

```text
p_action_rate = weight * norm(action_t - action_t_minus_1)^2
p_torque = weight * norm(torque)^2
p_joint_limit = weight * sum(max(0, abs(q) - soft_limit)^2)
```

## 训练课程设计

不要一开始就让机器人听完整摇滚歌曲并尝试蹦跳。训练应该像教学一样分阶段。

### 第 1 阶段：站稳

目标：在没有音乐条件下保持站立，能从小扰动中恢复。

训练内容：

- 随机初始姿态。
- 随机轻微外力推扰。
- 奖励 upright、base height、低速度、低能耗。

### 第 2 阶段：跟随简单正弦节拍

目标：让机器人学会在稳定前提下做周期性小幅动作。

训练内容：

- 输入固定 BPM 的相位信号。
- 目标动作为小幅点头或机身高度变化。
- 动作幅度很小，先不要求摇滚风格。

### 第 3 阶段：随机 BPM 和强度

目标：让策略适应不同速度和音乐能量。

训练内容：

- BPM 在 80 到 180 之间随机。
- 强度信号随机变化。
- 奖励动作幅度和强度匹配。

### 第 4 阶段：真实音乐特征

目标：用真实摇滚音乐的 beat、onset 和 energy 作为条件。

训练内容：

- 预先从音乐库提取特征，训练时随机抽取片段。
- 对 beat tracking 加入噪声，增强鲁棒性。
- 对音乐段落强弱做随机平滑，防止策略依赖单首歌。

### 第 5 阶段：观赏性增强

目标：在稳定基础上增加更像“摇滚”的动作。

训练内容：

- 加入参考动作库，例如人工设计的 headbang、sway、bounce 曲线。
- 使用 imitation reward 奖励低维动作特征接近参考，而不是强制每个关节完全复制。
- 对 downbeat 增加强调奖励。

## 推荐强化学习算法

第一版建议使用 PPO。

PPO 全称 Proximal Policy Optimization，是机器人连续控制里常用的强化学习算法。它的核心思想是：每次更新策略时不要改得太猛，避免今天学会站立，明天一更新又摔倒。

| 算法 | 是否推荐 | 理由 |
| --- | --- | --- |
| PPO | 推荐第一版 | 稳定、资料多、适合并行仿真 |
| SAC | 可作为后续对比 | 探索强，但机器人控制调参复杂一些 |
| TD3 | 可作为后续对比 | 适合连续动作，但训练流程不如 PPO 直观 |
| 行为克隆 | 可辅助 | 如果有参考动作，可先模仿再 RL 微调 |

## 神经网络结构

第一版可以使用简单 MLP：

```text
robot_state ----\
music_context ----> MLP -> action_mean
last_action ----/
```

策略输出动作均值，训练时加一点随机噪声用于探索，部署时使用均值动作。

更进一步可以加入历史编码：

```text
最近 N 帧音乐特征 -> Temporal Encoder -> music_embedding
机器人状态 + music_embedding -> policy
```

但第一版不建议一上来做复杂时序网络。先用 beat phase、energy、onset 等手工特征，可以更容易定位问题。

## 仿真随机化

为了让策略从 MuJoCo 迁移到真机，需要在训练时故意让仿真环境有变化。

| 随机项 | 目的 |
| --- | --- |
| 机身质量 | 适配模型质量误差 |
| 关节阻尼 | 适配电机和减速器差异 |
| 地面摩擦 | 适配不同地面 |
| 控制延迟 | 适配通信和底层控制延迟 |
| IMU 噪声 | 适配传感器噪声 |
| 音乐 beat 噪声 | 适配节拍检测错误 |

域随机化不是越大越好。太大时策略会变保守或学不动。建议从小范围开始，等基础动作稳定后逐步增加。

## 评估指标

| 指标 | 含义 |
| --- | --- |
| fall rate | 每 100 个 episode 中摔倒比例 |
| beat alignment error | 动作峰值和 beat 的平均时间差 |
| energy correlation | 动作能量和音乐能量的相关性 |
| joint limit violation | 关节接近或超过限制次数 |
| torque peak | 峰值力矩 |
| contact impact | 足端落地冲击 |
| visual score | 人工观看打分，判断是否像摇滚律动 |

节拍对齐可以这样计算：

```text
检测 observed_bob 的局部峰值
对每个 beat 找最近峰值
统计二者时间差的平均值和方差
```

## 真机部署策略

真机部署必须保守。

| 步骤 | 操作 |
| --- | --- |
| 离线回放 | 用真实音乐特征跑策略，但不连接真机，只记录动作目标 |
| 动作审查 | 检查关节角、速度、力矩估计和姿态目标是否越界 |
| 小幅模式 | 将 `action_scale` 降到训练值的 20% 到 40% |
| 低通滤波 | 对动作目标做平滑，避免高频抖动 |
| 急停监控 | 姿态、速度、足端接触异常时立刻回默认站姿或断开策略 |
| 逐步放大 | 每次只增加一个维度：幅度、速度、音乐强度或动作模式 |

## 最小算法闭环

第一版可以按这个闭环实现：

```text
1. 选 10 到 30 首摇滚音乐，离线提取 beat、BPM、onset、energy
2. 在 MuJoCo 中加载宇树机器人 XML/MJCF 模型
3. 构建 Gymnasium 风格环境
4. 状态 = 机器人本体状态 + 音乐条件 + 上一帧动作
5. 动作 = 默认站姿上的关节位置残差
6. 奖励 = 稳定 + 节拍对齐 + 能量匹配 - 安全/能耗/抖动惩罚
7. 使用 PPO 并行训练
8. 用未见过的歌曲评估
9. 导出仿真视频和指标
10. 通过安全审查后再进入真机小幅验证
```

## 常见误区

| 误区 | 正确做法 |
| --- | --- |
| 只奖励动作幅度大 | 同时奖励稳定、平滑和节拍对齐，否则会学出危险抖动 |
| 一开始就训练完整歌曲 | 先从固定 BPM 和简单相位开始 |
| 用单个音量值决定动作 | 使用 beat phase、onset、energy、段落强度组合 |
| 仿真成功就直接上真机 | 必须经过动作裁剪、低通滤波、急停和小幅验证 |
| 追求完全离地跳跃 | 第一版用视觉上的 bounce 感即可，真跳是高风险进阶目标 |
